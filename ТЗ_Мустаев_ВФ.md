## 1. ОБЩИЕ СВЕДЕНИЯ

**1.1. Наименование разработки**

Краткое наименование: «IT-ВыСотка Admin»

**1.2. Основание для разработки**

Настоящее техническое задание разработано в рамках выполнения выпускной квалификационной работы студента группы БПО-22-02 кафедры Вычислительной техники и инженерной кибернетики (ВТИК) Уфимского государственного нефтяного технического университета (УГНТУ).

## 2. ЦЕЛИ И ЗАДАЧИ СОЗДАНИЯ ВЕБ-ПРИЛОЖЕНИЯ

**2.1. Цель разработки**

Создание модуля администратора веб-приложения для комплексной автоматизации организационных процессов проведения чемпионата по программированию, обеспечивающего эффективное управление всеми этапами мероприятия.

**2.2. Задачи разработки**

1. Разработка модуля администратора
2. Реализация модуля управления участниками с возможностью фильтрации, поиска и экспорта данных
3. Создание модуля управления аудиториями и автоматической рассадки участников
4. Разработка модуля генерации персонализированных сертификатов в формате PDF
5. Реализация файлового менеджера для управления контентом веб-приложения
6. Создание модуля управления результатами чемпионата
7. Разработка модуля настроек веб-приложения и управления регистрацией
8. Реализация модуля резервного копирования базы данных
9. Разработка модуля управления волонтерами
10. Обеспечение безопасности модуля администратора через механизм аутентификации и авторизации

## 3. ТРЕБОВАНИЯ К ВЕБ-ПРИЛОЖЕНИЮ

**3.1. Требования к функциональным характеристикам**

**3.1.1. Модуль администратора**

Функция: Главная панель управления (Dashboard)

Описание: Отображение общей статистики и ключевых показателей веб-приложения

Требования: отображение общего количества зарегистрированных участников, разбивка участников по классам (8, 9, 10, 11), статистика по формату участия (индивидуальное/командное), количество созданных команд, статистика по заполненности эссе, информация о статусе регистрации, быстрый доступ к основным разделам веб-приложения.

**3.1.2. Управление участниками**

Функция: Просмотр и управление базой участников

Требования: просмотр списка всех зарегистрированных участников, фильтрация по классу (8, 9, 10, 11), фильтрация по формату участия (индивидуальное, командное, все), поиск по ФИО, email, школе, отображение информации (ФИО, email, класс, школа, формат участия, команда, эссе, место, сертификат), экспорт данных участников в формат Excel (.xlsx), установка места участнику (1-100), удаление участника, массовая рассылка напоминаний об эссе, массовая рассылка напоминаний участникам без команды, просмотр списка всех команд.

**3.1.3. Управление аудиториями**

Функция: Создание и управление аудиториями для проведения чемпионата

Требования: создание новой аудитории с указанием номера и вместимости, просмотр списка всех аудиторий, отображение номера аудитории, вместимости и количества занятых мест, редактирование параметров аудитории, удаление аудитории (с проверкой на отсутствие назначенных участников), валидация номера аудитории на уникальность, валидация вместимости на положительное число.

**3.1.4. Управление рассадкой участников**

Функция: Автоматическое и ручное распределение участников по аудиториям

Требования: автоматическая рассадка участников по аудиториям с учетом вместимости аудиторий, командного участия (члены команды должны быть в одной аудитории), оптимального заполнения аудиторий, просмотр текущей рассадки по аудиториям, отображение информации о каждом участнике в аудитории, ручное назначение участника в конкретную аудиторию, удаление участника из аудитории, просмотр списка нераспределенных участников, автоматическое добавление нераспределенных участников в свободные места, экспорт рассадки в формат Excel (.xlsx) с разбивкой по аудиториям, экспорт протоколов верификации, полная очистка рассадки для повторного распределения.

**3.1.5. Генерация сертификатов**

Функция: Создание и выдача персонализированных сертификатов участникам

Требования: загрузка шаблона сертификата в формате PDF, загрузка шрифта в формате TTF, интерактивная настройка позиции текста на сертификате (перемещение текста мышью, изменение размера шрифта 10-100 пт, выбор цвета текста RGB, предпросмотр в реальном времени), генерация PDF-сертификатов с использованием библиотеки PDF-lib, массовая выдача сертификатов с фильтрацией по классу (8, 9, 10, 11), по формату участия (индивидуальное, командное), по наличию места (только с местами, только без мест), по статусу сертификата (выдан/не выдан), отправка email-уведомлений участникам о готовности сертификата, просмотр списка участников с информацией о выданных сертификатах.

**3.1.6. Файловый менеджер**

Функция: Управление файлами и медиа-контентом веб-приложения

Требования: загрузка файлов с указанием типа (регламенты, галерея, спонсоры, прочее), просмотр списка всех загруженных файлов, фильтрация файлов по типу, отображение информации о файле (название, тип, размер, дата загрузки, URL для доступа), редактирование метаданных файла (название, описание, порядок отображения), удаление отдельных файлов, массовое удаление файлов, статистика по файлам (общее количество файлов, количество по типам, общий размер файлов), поддержка форматов: изображения (jpg, png, gif, webp), документы (pdf), видео.

**3.1.7. Управление настройками веб-приложения**

Функция: Конфигурирование параметров работы веб-приложения

Требования: управление статусом регистрации (открыта/закрыта), установка периода регистрации (дата начала и окончания), установка даты и времени проведения чемпионата, установка дедлайна для загрузки эссе, очистка данных для нового года с сохранением структуры базы данных, списка школ, административных аккаунтов, удаление при очистке: участников, команд, рассадки, сертификатов, файлов галереи, подтверждение операции очистки данных.

**3.1.8. Управление результатами чемпионата**

Функция: Создание и публикация результатов чемпионата

Требования: создание результатов вручную с указанием ФИО участника, класса, школы, места, формата участия, года проведения, автоматическое создание результатов из участников с присвоенными местами, просмотр результатов с фильтрацией по годам, редактирование существующих результатов, удаление результатов, отправка email-уведомлений победителям (1-3 места), публичный доступ к результатам в веб-приложении.

**3.1.9. Модуль резервного копирования**

Функция: Создание и управление резервными копиями базы данных

Требования: создание дампа базы данных PostgreSQL, автоматическое именование файлов с меткой времени, скачивание резервных копий, просмотр списка доступных резервных копий, отображение размера и даты создания копии, удаление старых резервных копий, хранение резервных копий в отдельной директории.

**3.1.10. Управление волонтерами**

Функция: Создание и управление аккаунтами волонтеров

Требования: создание новых аккаунтов волонтеров с указанием email и пароля, просмотр списка всех волонтеров, удаление волонтеров, изменение пароля волонтеров.

**3.1.11. Механизм безопасности**

Функция: Защита модуля администратора от несанкционированного доступа

Требования: middleware проверки прав администратора, проверка JWT токена доступа, автоматическое обновление токенов, централизованная обработка ошибок авторизации, разграничение доступа по ролям (admin - полный доступ ко всем функциям, volunteer - доступ только к панели волонтера, participant - доступ запрещен).

**3.2. Требования к надежности**

Веб-приложение обеспечивает непрерывную работу в течение всего периода проведения чемпионата. Время восстановления после сбоя не превышает 15 минут. Веб-приложение корректно обрабатывает ошибки и выводит информативные сообщения. Критические операции (удаление, очистка данных) требуют подтверждения. Автоматическое создание резервных копий базы данных. Возможность ручного создания резервной копии. Хранение резервных копий в защищенной директории. Валидация входных данных перед сохранением в базу данных.

**3.3. Требования к составу и параметрам технических средств**

Минимальные требования к серверу: процессор 2 ядра 2.0 GHz, оперативная память 4 GB RAM, дисковое пространство 20 GB, операционная система Linux (Ubuntu 20.04+, Debian 10+, CentOS 8+) или Windows Server 2016+, Docker 20.10+ и Docker Compose 2.0+.

Рекомендуемые требования к серверу: процессор 4 ядра 2.5 GHz, оперативная память 8 GB RAM, дисковое пространство 50 GB SSD, операционная система Linux (Ubuntu 22.04 LTS).

Минимальные требования к клиенту (рабочее место администратора): процессор 2 ядра 1.5 GHz, оперативная память 4 GB RAM, разрешение экрана 1366x768, веб-браузер Google Chrome 100+, Mozilla Firefox 100+, Safari 15+, Microsoft Edge 100+.

Рекомендуемые требования к клиенту: процессор 4 ядра 2.0 GHz, оперативная память 8 GB RAM, разрешение экрана 1920x1080 или выше, веб-браузер последняя версия Google Chrome или Mozilla Firefox.

**3.4. Требования к информационной и программной совместимости**

Backend: Node.js 20.0.0 или выше, Express.js 5.1.0, PostgreSQL 15.0 или выше, Sequelize ORM 6.35.0, JWT (jsonwebtoken 9.0.2), Nodemailer 6.9.0, PDF-lib 1.17.1, Multer 1.4.5-lts.1, ExcelJS 4.4.0, bcrypt 5.1.1, uuid 11.0.3.

Frontend: React 19.2.0, MobX 6.15.0, React Router 7.1.3, Axios 1.7.9.

Инфраструктура: Docker 20.10+, Docker Compose 2.0+, Nginx 1.25+ (Alpine).

Форматы данных: экспорт данных Microsoft Excel (.xlsx), импорт школ CSV (UTF-8), сертификаты PDF, шрифты TrueType Font (.ttf), изображения JPEG PNG GIF WebP, документы PDF, резервные копии SQL (PostgreSQL dump).

**3.5. Требования к методам и средствам защиты информации**

Использование JWT токенов для аутентификации. Access токен: срок действия 15 минут. Refresh токен: срок действия 30 дней. Хеширование паролей с использованием bcrypt (10 раундов). Проверка прав доступа на уровне middleware. Разграничение доступа по ролям (admin, volunteer, participant). Все пароли хранятся в хешированном виде. Токены хранятся в защищенных HTTP-only cookies. Валидация всех входных данных. Защита от SQL-инъекций через использование ORM Sequelize. Защита от XSS-атак через экранирование данных. CORS настроен для работы только с доверенными доменами. Загруженные файлы хранятся вне публичной директории. Доступ к файлам осуществляется через API с проверкой прав. Ограничение на размер загружаемых файлов (10 MB). Проверка типа загружаемых файлов. Уникальные имена файлов с использованием UUID.

**3.6. Требования к эргономике и технической эстетике**

Адаптивный дизайн для работы на различных устройствах. Единый стиль оформления всех страниц. Использование иконок для визуального представления функций. Цветовая индикация статусов (зеленый - успех, красный - ошибка, желтый - предупреждение). Боковое меню навигации с группировкой функций по разделам. Всплывающие уведомления (Toast) об успешных операциях и ошибках. Модальные окна для подтверждения критических операций. Индикаторы загрузки для длительных операций. Информативные сообщения об ошибках с указанием причины. Подсказки (tooltips) для элементов интерфейса. Поиск и фильтрация данных в таблицах. Сортировка данных по столбцам. Пагинация для больших списков данных.

Веб-приложение считается принятым, если реализованы все функции, указанные в разделе 3.1, веб-приложение работает стабильно без критических ошибок, интерфейс соответствует требованиям эргономики, обеспечена безопасность данных, подготовлена полная документация, веб-приложение развернуто и готово к использованию.
