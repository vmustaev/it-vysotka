## 2. ЦЕЛИ И ЗАДАЧИ СОЗДАНИЯ СИСТЕМЫ

### 2.1. Цель разработки

Создание административной части веб-приложения для комплексной автоматизации организационных процессов проведения чемпионата по программированию, обеспечивающей эффективное управление всеми этапами мероприятия.

### 2.2. Задачи разработки

1. Разработка административной панели с интуитивно понятным интерфейсом
2. Реализация модуля управления участниками с возможностью фильтрации, поиска и экспорта данных
3. Создание системы управления аудиториями и автоматической рассадки участников
4. Разработка модуля генерации персонализированных сертификатов в формате PDF
5. Реализация файлового менеджера для управления контентом сайта
6. Создание системы управления результатами чемпионата
7. Разработка модуля настроек системы и управления регистрацией
8. Реализация системы резервного копирования базы данных
9. Создание панели волонтера для контроля присутствия участников
10. Обеспечение безопасности административной части через систему аутентификации и авторизации

## 3. ТРЕБОВАНИЯ К СИСТЕМЕ

### 3.1. Требования к функциональным характеристикам

#### 3.1.1. Административная панель

**Функция:** Главная панель управления (Dashboard)

**Описание:** Отображение общей статистики и ключевых показателей системы

**Требования:**
- Отображение общего количества зарегистрированных участников
- Разбивка участников по классам (8, 9, 10, 11)
- Статистика по формату участия (индивидуальное/командное)
- Количество созданных команд
- Статистика по заполненности эссе
- Информация о статусе регистрации
- Быстрый доступ к основным разделам системы

#### 3.1.2. Управление участниками

**Функция:** Просмотр и управление базой участников

**Требования:**
- Просмотр списка всех зарегистрированных участников
- Фильтрация по классу (8, 9, 10, 11)
- Фильтрация по формату участия (индивидуальное, командное, все)
- Поиск по ФИО, email, школе
- Отображение информации: ФИО, email, класс, школа, формат участия, команда, эссе, место, сертификат
- Экспорт данных участников в формат Excel (.xlsx)
- Установка места участнику (1-100)
- Удаление участника из системы
- Массовая рассылка напоминаний об эссе
- Массовая рассылка напоминаний участникам без команды
- Просмотр истории изменений профиля участника

#### 3.1.3. Управление командами

**Функция:** Просмотр информации о командах

**Требования:**
- Просмотр списка всех созданных команд
- Отображение названия команды
- Отображение капитана команды
- Отображение списка участников команды (до 3 человек)
- Отображение даты создания команды

#### 3.1.4. Управление аудиториями

**Функция:** Создание и управление аудиториями для проведения чемпионата

**Требования:**
- Создание новой аудитории с указанием номера и вместимости
- Просмотр списка всех аудиторий
- Отображение номера аудитории, вместимости и количества занятых мест
- Редактирование параметров аудитории
- Удаление аудитории (с проверкой на отсутствие назначенных участников)
- Валидация: номер аудитории должен быть уникальным
- Валидация: вместимость должна быть положительным числом

#### 3.1.5. Управление рассадкой участников

**Функция:** Автоматическое и ручное распределение участников по аудиториям

**Требования:**
- Автоматическая рассадка участников по аудиториям с учетом:
  - Вместимости аудиторий
  - Командного участия (члены команды должны быть в одной аудитории)
  - Оптимального заполнения аудиторий
- Просмотр текущей рассадки по аудиториям
- Отображение информации о каждом участнике в аудитории
- Ручное назначение участника в конкретную аудиторию
- Удаление участника из аудитории
- Просмотр списка нераспределенных участников
- Автоматическое добавление нераспределенных участников в свободные места
- Экспорт рассадки в формат Excel (.xlsx) с разбивкой по аудиториям
- Полная очистка рассадки для повторного распределения

#### 3.1.6. Генерация сертификатов

**Функция:** Создание и выдача персонализированных сертификатов участникам

**Требования:**
- Загрузка шаблона сертификата в формате PDF
- Загрузка пользовательского шрифта в формате TTF
- Интерактивная настройка позиции текста на сертификате:
  - Перемещение текста мышью
  - Изменение размера шрифта (10-100 пт)
  - Выбор цвета текста (RGB)
  - Предпросмотр в реальном времени
- Генерация PDF-сертификатов с использованием библиотеки PDF-lib
- Массовая выдача сертификатов с фильтрацией:
  - По классу (8, 9, 10, 11)
  - По формату участия (индивидуальное, командное)
  - По наличию места (только с местами, только без мест, все)
  - По статусу сертификата (выдан/не выдан)
- Отправка email-уведомлений участникам о готовности сертификата
- Просмотр списка участников с информацией о выданных сертификатах
- Скачивание сертификатов участниками из личного кабинета

#### 3.1.7. Файловый менеджер

**Функция:** Управление файлами и медиа-контентом сайта

**Требования:**
- Загрузка файлов с указанием типа:
  - Регламенты (regulations)
  - Галерея (gallery)
  - Спонсоры (sponsors)
  - Прочее (other)
- Просмотр списка всех загруженных файлов
- Фильтрация файлов по типу
- Отображение информации о файле:
  - Название
  - Тип
  - Размер
  - Дата загрузки
  - URL для доступа
- Редактирование метаданных файла (название, описание, порядок отображения)
- Удаление файлов
- Статистика по файлам:
  - Общее количество файлов
  - Количество по типам
  - Общий размер файлов
- Поддержка форматов: изображения (jpg, png, gif, webp), документы (pdf), видео

#### 3.1.8. Управление настройками системы

**Функция:** Конфигурирование параметров работы системы

**Требования:**
- Управление статусом регистрации (открыта/закрыта)
- Установка периода регистрации (дата начала и окончания)
- Установка даты и времени проведения чемпионата
- Установка дедлайна для загрузки эссе
- Очистка данных для нового года с сохранением:
  - Структуры базы данных
  - Списка школ
  - Административных аккаунтов
- Удаление при очистке:
  - Участников
  - Команд
  - Рассадки
  - Сертификатов
  - Файлов галереи
- Подтверждение операции очистки данных

#### 3.1.9. Управление результатами чемпионата

**Функция:** Создание и публикация результатов чемпионата

**Требования:**
- Создание результатов вручную с указанием:
  - ФИО участника
  - Класса
  - Школы
  - Места
  - Формата участия
  - Года проведения
- Автоматическое создание результатов из участников с присвоенными местами
- Просмотр результатов с фильтрацией по годам
- Редактирование существующих результатов
- Удаление результатов
- Отправка email-уведомлений победителям (1-3 места)
- Публичный доступ к результатам на сайте

#### 3.1.10. Система резервного копирования

**Функция:** Создание и управление резервными копиями базы данных

**Требования:**
- Создание дампа базы данных PostgreSQL
- Автоматическое именование файлов с меткой времени
- Скачивание резервных копий
- Просмотр списка доступных резервных копий
- Отображение размера и даты создания копии
- Удаление старых резервных копий
- Хранение резервных копий в отдельной директории

#### 3.1.11. Панель волонтера

**Функция:** Контроль присутствия участников на чемпионате

**Требования:**
- Просмотр списка участников с рассадкой по аудиториям
- Фильтрация участников по аудиториям
- Поиск участников по ФИО, школе, команде
- Отметка присутствия отдельного участника
- Массовая отметка присутствия для всей аудитории
- Отображение статистики присутствия:
  - Общее количество участников
  - Количество присутствующих
  - Количество отсутствующих
  - Процент явки
- Статистика по аудиториям
- История отметок присутствия
- Отображение времени отметки

#### 3.1.12. Система безопасности

**Функция:** Защита административной части от несанкционированного доступа

**Требования:**
- Middleware проверки прав администратора
- Middleware проверки прав волонтера
- Проверка JWT токена доступа
- Автоматическое обновление токенов
- Централизованная обработка ошибок авторизации
- Разграничение доступа по ролям:
  - admin: полный доступ ко всем функциям
  - volunteer: доступ только к панели волонтера
  - participant: доступ запрещен

### 3.2. Требования к надежности

#### 3.2.1. Обеспечение надежности

- Система должна обеспечивать непрерывную работу в течение всего периода проведения чемпионата
- Время восстановления после сбоя не должно превышать 15 минут
- Система должна корректно обрабатывать все ошибки и выводить информативные сообщения
- Все критические операции (удаление, очистка данных) должны требовать подтверждения

#### 3.2.2. Контроль и восстановление данных

- Автоматическое создание резервных копий базы данных
- Возможность ручного создания резервной копии в любой момент
- Хранение резервных копий в защищенной директории
- Валидация всех входных данных перед сохранением в базу данных

### 3.5. Требования к структуре и организации данных

#### 3.5.1. База данных

Система использует реляционную СУБД PostgreSQL 15 со следующими основными таблицами:

**users** - Пользователи системы
- id (UUID, PK)
- email (VARCHAR, UNIQUE)
- password (VARCHAR, хешированный)
- firstName (VARCHAR)
- lastName (VARCHAR)
- patronymic (VARCHAR)
- grade (INTEGER: 8, 9, 10, 11)
- schoolId (INTEGER, FK)
- role (ENUM: 'participant', 'admin', 'volunteer')
- participationFormat (ENUM: 'individual', 'team')
- teamId (UUID, FK, nullable)
- essayUrl (TEXT, nullable)
- place (INTEGER, nullable)
- certificateIssued (BOOLEAN, default: false)
- isActivated (BOOLEAN, default: false)
- activationLink (VARCHAR, nullable)
- createdAt (TIMESTAMP)
- updatedAt (TIMESTAMP)

**teams** - Команды участников
- id (UUID, PK)
- name (VARCHAR, UNIQUE)
- captainId (UUID, FK)
- inviteToken (VARCHAR, UNIQUE)
- createdAt (TIMESTAMP)
- updatedAt (TIMESTAMP)

**schools** - Образовательные учреждения
- id (INTEGER, PK, AUTO_INCREMENT)
- region (VARCHAR)
- city (VARCHAR)
- name (VARCHAR)
- createdAt (TIMESTAMP)
- updatedAt (TIMESTAMP)

**rooms** - Аудитории
- id (INTEGER, PK, AUTO_INCREMENT)
- roomNumber (VARCHAR, UNIQUE)
- capacity (INTEGER)
- createdAt (TIMESTAMP)
- updatedAt (TIMESTAMP)

**seating_assignments** - Рассадка участников
- id (INTEGER, PK, AUTO_INCREMENT)
- userId (UUID, FK, UNIQUE)
- roomId (INTEGER, FK)
- createdAt (TIMESTAMP)
- updatedAt (TIMESTAMP)

**settings** - Настройки системы
- id (INTEGER, PK, AUTO_INCREMENT)
- registrationOpen (BOOLEAN, default: true)
- registrationStartDate (DATE, nullable)
- registrationEndDate (DATE, nullable)
- championshipDate (TIMESTAMP, nullable)
- essayDeadline (TIMESTAMP, nullable)
- certificateTemplate (TEXT, nullable)
- certificateFont (TEXT, nullable)
- certificateSettings (JSON, nullable)
- createdAt (TIMESTAMP)
- updatedAt (TIMESTAMP)

**files** - Файлы системы
- id (INTEGER, PK, AUTO_INCREMENT)
- name (VARCHAR)
- originalName (VARCHAR)
- path (VARCHAR)
- type (ENUM: 'regulations', 'gallery', 'sponsors', 'other')
- size (INTEGER)
- mimeType (VARCHAR)
- description (TEXT, nullable)
- displayOrder (INTEGER, default: 0)
- createdAt (TIMESTAMP)
- updatedAt (TIMESTAMP)

**championship_results** - Результаты чемпионата
- id (INTEGER, PK, AUTO_INCREMENT)
- fullName (VARCHAR)
- grade (INTEGER)
- school (VARCHAR)
- place (INTEGER)
- participationFormat (ENUM: 'individual', 'team')
- year (INTEGER)
- createdAt (TIMESTAMP)
- updatedAt (TIMESTAMP)

**tokens** - Токены (refresh, reset, activation)
- id (INTEGER, PK, AUTO_INCREMENT)
- userId (UUID, FK)
- refreshToken (TEXT, nullable)
- resetToken (TEXT, nullable)
- resetTokenExpiry (TIMESTAMP, nullable)
- createdAt (TIMESTAMP)
- updatedAt (TIMESTAMP)

**attendance_history** - История отметок присутствия
- id (INTEGER, PK, AUTO_INCREMENT)
- userId (UUID, FK)
- isPresent (BOOLEAN)
- markedBy (UUID, FK)
- markedAt (TIMESTAMP)
- createdAt (TIMESTAMP)
- updatedAt (TIMESTAMP)

**profile_history** - История изменений профиля
- id (INTEGER, PK, AUTO_INCREMENT)
- userId (UUID, FK)
- changes (JSON)
- changedBy (UUID, FK)
- changedAt (TIMESTAMP)
- createdAt (TIMESTAMP)
- updatedAt (TIMESTAMP)

### 3.6. Требования к методам и средствам защиты информации

#### 3.6.1. Аутентификация и авторизация

- Использование JWT токенов для аутентификации
- Access токен: срок действия 15 минут
- Refresh токен: срок действия 30 дней
- Хеширование паролей с использованием bcrypt (10 раундов)
- Проверка прав доступа на уровне middleware
- Разграничение доступа по ролям (admin, volunteer, participant)

#### 3.6.2. Защита данных

- Все пароли хранятся в хешированном виде
- Токены хранятся в защищенных HTTP-only cookies
- Валидация всех входных данных
- Защита от SQL-инъекций через использование ORM Sequelize
- Защита от XSS-атак через экранирование данных
- CORS настроен для работы только с доверенными доменами

#### 3.6.3. Защита файлов

- Загруженные файлы хранятся вне публичной директории
- Доступ к файлам осуществляется через API с проверкой прав
- Ограничение на размер загружаемых файлов (10 MB)
- Проверка типа загружаемых файлов
- Уникальные имена файлов с использованием UUID
